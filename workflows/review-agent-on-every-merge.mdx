The pattern is one line of policy: **no PR merges without passing the review gate**. The gate's three questions — *Can we trust what changed? Does it meet our standards? Is it safe to merge?* — get answered by some combination of automated review, type-check, tests, and human reading. The composition varies by team; the policy doesn't.

The recipe is a GitHub branch-protection rule plus a webhook:

```text
Branch protection on `main`:
  ☑ Require pull request before merging
  ☑ Require status checks to pass:
      - review-agent (CodeRabbit / Greptile / etc.)
      - lint-and-typecheck
      - tests
  ☑ Require approvals (1+ human)
```

The pattern's wedge is the **independence** of the reviewer from the author. When a human writes the diff, a different human reviews. When an agent writes the diff, a different agent (or a human, or both) reviews. The reviewer never trusts the author's assertion that the change is correct — it verifies.

**Use for** any repo where agents will routinely write code. The cost is roughly one tool subscription and ~30 minutes of CI configuration; the payoff is the discharge mechanism for [review debt](/knowledge-base/glossary/review-debt).

**Avoid for** throwaway prototypes and personal exploration repos. The gate is policy infrastructure; it shouldn't apply to playgrounds.

This is the universal pattern. Tool-specific implementations include [`coderabbit-agent-mode`](/workflows/coderabbit-agent-mode), [`claude-code-github-action`](/workflows/claude-code-github-action), and standalone review tools — see also [/the-review-gate](/the-review-gate) for the framework these all implement.
