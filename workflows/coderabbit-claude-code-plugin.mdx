The plugin makes CodeRabbit invokable from inside Claude Code as a slash command. Same review engine as the GitHub app, different invocation surface: instead of opening a PR and waiting for the bot, you run the reviewer against your working tree, your last commit, or a branch comparison, *before* the PR exists.

## Why this matters

The GitHub-app version of CodeRabbit reviews after the PR is opened. By the time the findings arrive, you've already committed, pushed, and named the change publicly. The plugin shifts review *left* — into the loop where the agent (or you) is still iterating. Findings get addressed in the same session that wrote the code, not in a PR ping six minutes later.

This is especially load-bearing for agent-authored work. The agent writes a diff, you run `/coderabbit:code-review`, the findings come back inside the same Claude Code context, and the agent can act on them immediately. No PR ceremony, no context switch.

## The four scopes

- `/coderabbit:code-review` — review everything in the working tree (committed + uncommitted)
- `/coderabbit:code-review committed` — only what's been committed locally
- `/coderabbit:code-review uncommitted` — only the dirty working tree (good for "before I commit this, sanity check")
- `/coderabbit:code-review --base main` — compare against a branch (PR-shaped review without the PR)

## Use it

The plugin pairs well with [Review-agent-on-every-merge](/workflows/review-agent-on-every-merge) for the PR-time gate, and with [CodeRabbit Agent Mode](/workflows/coderabbit-agent-mode) for the in-PR multi-agent pipeline. The plugin's wedge is the *pre-PR* slot none of those cover.

**Use for** agent-authored PRs where you want the review feedback before the PR opens; tight inner loops where you'd rather get findings in the same Claude Code session than as a delayed bot comment.

**Avoid for** trivial changes (config bumps, typo fixes) — running the full reviewer is overhead the change doesn't earn back.
