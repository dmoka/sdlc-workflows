A full sprint-as-skills framework by Garry Tan (YC President). 23 opinionated Claude Code slash commands that play specialist roles, CEO, designer, eng manager, reviewer, QA lead, security officer, release engineer. Each command is a Markdown file; nothing is magic. MIT licensed, 93k+ stars.

The thesis is that *prompting alone* leaves too much on the table. Structured *roles* per sprint phase produce shippable PRs from a single developer.

## The sprint

The commands run in the order a sprint runs:

> **Think → Plan → Build → Review → Test → Ship → Reflect**

Each phase feeds the next. `/office-hours` writes the design doc that `/plan-ceo-review` reads. `/plan-eng-review` writes the test plan that `/qa` picks up. `/review` catches bugs that `/ship` verifies are fixed. Nothing falls through the cracks because every step knows what came before it.

## 5-step Quick Start

Don't try all 23 commands at once. The README explicitly says:

1. **Install gstack** (30 seconds, paste the install command into Claude Code)
2. **`/office-hours`**, describe what you're building, let the agent interrogate the pain points
3. **`/plan-ceo-review`**, strategic scope challenge on any feature idea
4. **`/review`**, multi-aspect code review on any branch with changes
5. **`/qa https://staging`**, real browser, real clicks, finds and fixes the bugs your linter can't

> "Stop there. You'll know if this is for you."

(Full catalog of 23 commands + 8 power tools in the README.)

**Use for** founders and tech leads who want a structured sprint per feature, with rigorous review + QA + release automation. The opinionation is the point, gstack assumes you want production-quality output, not a brainstorming buddy.

**Avoid for** one-line fixes or throwaway scripts. The install + 7-phase sprint overhead overshoots the work.

## Why it works

Three structural choices the README hammers home:

- **Roles, not prompts.** Each command IS a specialist; there's no "be a CEO and review this" prompt-engineering. The role is baked into the command markdown.
- **Skills hand off.** Output of one skill is input to the next. No re-explaining context across phases.
- **Real verification.** `/review` finds bugs, `/qa` clicks through a browser, `/cso` runs security audits. Verification isn't an afterthought, it's a phase.

gstack is opinionated by design. You can fork and adapt the individual `.md` files; the install is `git clone` + run `setup`, nothing vendored, nothing proprietary.
