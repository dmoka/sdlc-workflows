A specialised evaluator-optimizer loop from Andrej Karpathy: give an AI agent a small but real LLM training setup, point it at a hard scalar metric (validation bits-per-byte), and let it iterate overnight. The agent modifies the code, trains for a fixed 5-minute wall-clock budget, checks if the metric improved, keeps or discards, and repeats. ~100 experiments per overnight run.

Three files do the work:

- **`prepare.py`**, fixed constants + one-time data prep + runtime utilities. The agent never touches this.
- **`train.py`**, the full GPT model, optimizer, and training loop. The single file the agent edits. Architecture, hyperparameters, optimizer choice, everything is fair game.
- **`program.md`**, instructions to the agent. The human's job is to iterate on *this* file, not the Python. "You are programming the `program.md` Markdown files," as Karpathy puts it. It's a lightweight skill.

The design choices matter:

- **Single file to modify** keeps diffs reviewable and scope manageable.
- **Fixed 5-minute time budget** makes experiments directly comparable regardless of what the agent changed (model size, batch size, architecture). It also means autoresearch finds the most-optimal config *for your specific hardware*.
- **One scalar metric** (`val_bpb`), vocab-size-independent, so architectural changes are fairly compared.
- **Self-contained**, one GPU, one file, one metric.

## The generalisation

The repo is ML-specific, but the *shape* is universal. Anywhere you have:

- a **single well-scoped artifact** the agent is allowed to mutate freely,
- a **single scalar metric** that defines success,
- **independent iterations** (each run is from-scratch, not incremental),
- and a **fixed time budget** per run —

the autoresearch pattern works. Latency optimisation. Compression. Eval-suite tuning. Any closed-loop search problem where you can measure goodness as a number.

## Run it

> ⚠ Runs unattended overnight with permissions skipped. Use a fresh worktree, container, or dedicated machine, not your main filesystem.

Spin up Claude Code / Codex / your agent of choice in the repo, disable permissions, then prompt:

```text
Hi, have a look at program.md and let's kick off a new experiment.
Let's do the setup first.
```

You wake up in the morning to a log of experiments and, if `program.md` is well-tuned, a better model.

**Use for** ML training search; latency / compression / efficiency search; any closed-loop optimization where the metric is unambiguous.

**Avoid for** work that isn't measurable as one number, or where iterations share state. For multi-file features and refactors, use [Ralph](/workflows/ralph-wiggum-loop) or [PIV](/workflows/piv-loop) instead.
