The cheapest discipline that prevents large unproductive code dumps. Before the agent writes any code, it produces a written plan: scope, approach, files to touch, success criteria. The user approves in plain English. Only then does the agent execute.

The recipe is a single instruction prepended to the task prompt:

```text
First message rule: output PLAN ONLY. List scope, approach, files,
success criteria. Do NOT write code until I reply with "APPROVED".
```

That's it. The technique converges across multiple frameworks under different names — **Plan & Act** in Cline, **Plan mode** in Claude Code (`shift+tab`), **architect mode** in Aider, **Specify → Clarify → Plan** in Spec Kit. They all share the same beat: separate the *what* from the *how*, gate the second on a human OK.

The plan-first discipline does three things at once. It surfaces misunderstandings before they cost code. It gives you something concrete to push back on ("don't touch the auth module"). And it gives the agent something concrete to follow when it later loses thread — re-reading its own plan beats re-deriving it.

**Use for** anything non-trivial. The plan step costs ~1-2 minutes of conversation and saves an order of magnitude more by preventing rework.

**Avoid for** mechanical or trivially-bounded tasks: formatting passes, automated test stubs, dependency upgrades. The plan there is the task name, and the gate adds friction with no payoff.
