The decomposition pattern. A coordinator (orchestrator, supervisor, parent agent) reads the task, splits it into N independent subtasks, dispatches them to N workers in parallel, then merges the results into a single deliverable.

Three things make fan-out work where it works:

1. **Independence.** The subtasks must not depend on each other's intermediate state. If subtask 3 needs subtask 1's output, you've serialized the work; fan-out gives you no parallelism.
2. **Isolation.** Each worker gets its own context window. They can't see each other's notes, scratch space, or partial results. The orchestrator is the only one with the global view.
3. **Mergeability.** The subtask outputs have to be reconcilable. A refactor across N files is mergeable; debating which framework to use across N agents is not.

```text
ORCHESTRATOR prompt:
  Decompose this task into 3-7 independent subtasks. Each subtask
  must be doable without knowing what the others are doing. Output
  the subtask list. Then dispatch.

WORKER prompt (one per subtask):
  Do exactly this subtask, nothing else. Output only the result.

ORCHESTRATOR (merge phase):
  Reconcile these N outputs into a single deliverable.
```

**Use for** at-scale refactors, parallel test generation, debugging-by-hypothesis (each worker tests one hypothesis), batched dependency upgrades across N repos.

**Avoid for** sequential reasoning, tightly-coupled changes (two workers editing the same file), or tasks where global context matters per-worker.
