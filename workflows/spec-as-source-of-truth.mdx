The discipline of treating the **spec**, not the code, as the canonical artifact. Code is a derivation. When behavior needs to change, the spec changes first; the implementation changes to match. The order of operations matters — edit the spec before the code, every time.

The shape:

```text
specs/
  features/
    auth-2fa.md          # the spec for this feature
    auth-2fa.plan.md     # the task plan derived from the spec
  invariants.md          # what must remain true across all features
```

A canonical workflow:

```text
1. Edit (or write) `specs/features/<feature>.md`
2. Generate a plan from the spec — agent decomposes into tasks
3. Human approves the plan
4. Agent implements task-by-task, referencing the spec
5. Tests are derived from the spec, not from intuition
6. Code reviewer asks: "does this code match the spec?"
```

The wedge over "just write code" is that the spec is human-readable, the spec is a single artifact you can argue about, and the spec encodes intent in a form that survives refactors. The wedge over "write a design doc that gets stale" is that the spec is *load-bearing for the implementation* — if it gets stale, the next code change forces you to refresh it.

**Use for** features with real behavior contracts (auth, billing, ingestion, anything with multiple consumers). Especially valuable for agentic workflows: the spec is the unambiguous instruction the agent reads, not the implicit-knowledge cloud a human team might tolerate.

**Avoid for** prototypes (the spec overhead exceeds the prototype's value) and UI work that's design-driven (the design *is* the spec; writing a separate text spec doubles work).

Tool-specific implementations: [`spec-kit-specify`](/workflows/spec-kit-specify) (GitHub's CLI), Kiro (an IDE built around this), [`plan-approval-gate`](/workflows/plan-approval-gate) (minimal version, no separate spec file), [`ai-dlc`](/workflows/ai-dlc) (AWS's full SDLC variant).
